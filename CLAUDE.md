## You are an agent running in this repo

This repo is the **adaptation analysis** for the Postman CSE take-home. The
canonical pattern lives in the companion repo
[`jr-cse-payments-postman-onboarding`](https://github.com/JeremiahJRRoss/jr-cse-payments-postman-onboarding).

different characteristics — ECS+ALB compute, mTLS+JWT auth — and document the
diff in `ADAPTATION.md`.

## Critical constraints (same as companion repo)

1. **Mixed infrastructure** — this service is ECS+ALB; contrast with Payments' Lambda+API Gateway.
2. **Mixed CI/CD** — GitHub Actions here; one team on GitLab CI won't switch.
3. **Inconsistent OpenAPI coverage** across the org. This service's spec is current.
4. **Platform team stretched thin** — no new tools.
5. **Governance, not utilization** — catalog story.

different compute AND different auth, the pattern scales.** That's the
deliverable this repo demonstrates.

## Refusals

- **Don't re-author the workflow from scratch.** Copy from companion; change
  only the per-service inputs. The pattern-transfer story is undermined by
  parallel authorship.
- **Don't pad ADAPTATION.md with restated pattern explanation.** This repo is
  the delta. If you find yourself explaining what the action chain does, stop
  and point at companion's README.
- **Don't inflate the diff to look thorough.** If 6 lines changed, say 6.
- **Don't treat mTLS as a bug in the action.** The spec doesn't declare mTLS
  in `securitySchemes` — only in `info.description` text. The action correctly
  follows the spec. mTLS is customer-side configuration.
- **Don't recommend tools the platform team would adopt.**
- **Don't push to remote without showing the user the change first.**
